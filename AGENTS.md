# Project AGENTS

本文件是 `track_geopolitics` 的项目级工作规则，优先于全局偏好。

## 启动

- 开始较大任务前，先读最小必要集合：`README.md`、`.claude/CLAUDE.md`，以及相关子目录说明。
- 把 `.claude/` 中的规则视为参考资料，按当前 Codex 工具环境执行，不假设 Claude 专用工具存在。

## 沟通

- 默认使用中文。
- 先行动，后汇报；只有在存在隐藏风险或不可逆后果时才提问。
- 回答简洁，优先说明结果、风险、下一步。

## 文件与数据

- 先检查再编辑，避免凭印象批量修改 CSV、JSON、Markdown。
- 临时脚本放在 `_tmp/`，用完删除；不要在仓库根目录散落临时文件。
- 对结构化数据文件，优先保持原有字段顺序、编码、分隔符、日期格式不变。
- 涉及 CSV 编码转换、批量回填、批量替换、日期格式调整时，必须先做样本核验，再扩大到全表。

## Git Checkpoint Rules

- 任何会修改结构化数据的任务，在动手前先看一次 `git status --short`。
- 满足以下任一条件时，必须先做一次 checkpoint commit，再继续编辑：
  - 会批量修改 CSV、JSON、TSV、Excel 导出结果。
  - 会改动 20 行以上结构化数据。
  - 会影响编码、换行符、分隔符、引号、日期格式。
  - 会运行批量回填、清洗、去重、格式转换脚本。
- 完成一个可验证的小阶段后，应立即再做一次 checkpoint commit，不要把多个独立修复混在一个提交里。
- 如果工作区已有用户未提交改动，不得擅自提交；先说明哪些文件已脏、当前任务准备如何避开或处理。
- 做 checkpoint commit 前，先做最小验证：抽样检查关键行、确认文件可正常读取、确认编码未损坏。
- 涉及高风险数据修复时，优先“小步提交”，不要一次提交多个清单或多个数据源。

## 本项目特别注意

- `docs/research/china-counter-sanctions/data/US/` 下的清单文件默认应保持 `UTF-8 with BOM`，除非文件本身已有其他明确约定。
- 修改制裁清单时，优先保留源清单与汇总表之间的一致口径，并明确说明哪些表已同步、哪些尚未同步。
- 统计“美国制裁了多少中国实体”时，必须区分：
  - 按清单累计条目数
  - 跨清单去重后的实体数
  - 数据截止日期
- 使用“截至今天/最新”这类表述时，要给出绝对日期，例如 `截至 2026-04-14`。
